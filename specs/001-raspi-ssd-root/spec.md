# Feature Specification: Перенос rootfs Raspberry Pi на SSD

**Feature Branch**: `001-raspi-ssd-root`
**Created**: 2025-02-14
**Status**: Draft
**Input**: User description: "Перенос rootfs Raspberry Pi на SSD при загрузке с microSD. Создание Ansible роли для автоматизации. Тестирование на хосте leha (10.0.1.104)"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Подготовка SSD к использованию (Priority: P1)

Системный администратор имеет Raspberry Pi с установленной ОС на microSD карте и хочет повысить производительность системы за счёт использования SSD вместо медленной карты памяти для хранения системных файлов, логов и данных.

**Why this priority**: Это базовая функциональность без которой весь rest функции не работает. Без подготовки SSD (разметка и форматирование) невозможно перенести на него файловую систему.

**Independent Test**: Можно проверить независимо - после выполнения роль должна сообщить что SSD отформатирован и готов к использованию, можно проверить через `lsblk` и `df -h`.

**Acceptance Scenarios**:

1. **Given** Raspberry Pi с подключенным SSD через USB, **When** выполняется Ansible роль, **Then** SSD автоматически определяется и инициализируется
2. **Given** SSD без разделов, **When** роль выполняет разметку диска, **Then** создаётся один раздел на весь объём диска с файловой системой ext4
3. **Given** SSD с существующими данными, **When** роль обнаруживает раздел, **Then** администратор получает предупреждение и подтверждение перед форматированием

---

### User Story 2 - Перенос корневой файловой системы (Priority: P1)

Системный администратор хочет перенести работающую систему с microSD на SSD чтобы снизить износ карты памяти и увеличить скорость операций ввода-вывода.

**Why this priority**: Это основная цель функции - перенос rootfs обеспечивает все преимущества SSD (скорость, надёжность).

**Independent Test**: После выполнения можно проверить монтирование через `df -h` - корень `/` должен быть на SSD (`/dev/sdX1`), а `/boot` остаётся на microSD (`/dev/mmcblk0p1`).

**Acceptance Scenarios**:

1. **Given** Подготовленный SSD с ext4, **When** роль выполняет копирование, **Then** все файлы из rootfs переносятся на SSD с сохранением прав доступа, владельцев и символических ссылок
2. **Given** Процесс копирования завершён, **When** проверяется целостность, **Then** размер использованного пространства на SSD совпадает с исходным на microSD
3. **Given** Скопированная система на SSD, **When** происходит перезагрузка, **Then** система загружается с SSD как корневого раздела

---

### User Story 3 - Настройка конфигурации для загрузки с SSD (Priority: P1)

Системный администратор хочет чтобы Raspberry Pi автоматически загружал систему с SSD при каждом старте, без необходимости ручной настройки параметров ядра.

**Why this priority**: Без правильной настройки конфигурации система не сможет загрузиться с SSD.

**Independent Test**: После настройки можно проверить содержимое `/mnt/ssd/etc/fstab` и `/boot/cmdline.txt` - они должны содержать UUID SSD раздела в параметрах монтирования корня.

**Acceptance Scenarios**:

1. **Given** Скопированная система на SSD, **When** роль обновляет fstab, **Then** строка монтирования корня `/` использует UUID SSD раздела вместо PARTUUID
2. **Given** Обновлённый fstab, **When** проверяется конфигурация, **Then** раздел `/boot` продолжает указывать на microSD карту
3. **Given** Скопированная система, **When** роль обновляет cmdline.txt, **Then** параметр `root=` указывает на UUID SSD раздела

---

### User Story 4 - Верификация успешного переноса (Priority: P2)

Системный администратор хочет получить подтверждение что перенос прошёл успешно и система работает с SSD.

**Why this priority**: Важно убедиться что функция работает как ожидается, но проверка может быть выполнена вручную после автоматического переноса.

**Independent Test**: Можно запустить отдельно от основного сценария - проверить что корень смонтирован с SSD и система работает стабильно.

**Acceptance Scenarios**:

1. **Given** Перезагружённая система после переноса, **When** выполняется команда `df -h`, **Then** корень `/` отображается как смонтированный с устройства `/dev/sdX1` соответствующего SSD
2. **Given** Система загружена с SSD, **When** выполняются операции ввода-вывода (чтение/запись файлов), **Then** наблюдается повышенная скорость по сравнению с microSD
3. **Given** Работающая система на SSD, **When** проверяется запись логов и временных файлов, **Then** все изменения сохраняются на SSD а не на microSD карте

---

### User Story 5 - Интеграция с существующими Ansible ролями (Priority: P2)

Системный администратор хочет использовать эту новую роль совместно с существующими ролями кластера для единой точки управления конфигурацией всех узлов.

**Why this priority**: Это удобство использования и согласованность с проектной структурой, но не влияет на базовую функциональность переноса.

**Independent Test**: Можно выполнить роль из другого playbook или совместно с другими ролями - она должна корректно работать в контексте большего play.

**Acceptance Scenarios**:

1. **Given** Существующие роли проекта (common, locale, cluster_setup), **When** создаётся новая роль ssd_rootfs, **Then** она следует той же структуре директорий (tasks, handlers, templates, vars)
2. **Given** Новая роль ssd_rootfs, **When** она выполняется на хосте leha (10.0.1.104), **Then** роль корректно применяется к конкретному хосту из inventory
3. **Given** Роль выполнена успешно, **When** запускается повторно, **Then** роль является идемпотентной и не выполняет лишних действий

---

### Edge Cases

- **SSD не обнаружен**: Что происходит если SSD не подключен или не определяется системой? Роль должна сообщить об ошибке и прекратить выполнение.
- **Недостаточно места на SSD**: Что если размер SSD меньше чем текущее использование на microSD? Роль должна проверить свободное место и предупредить администратора.
- **Несколько SSD дисков**: Какой диск выбрать если подключено несколько устройств хранения? Роль должна позволять явно указать целевой диск или использовать первый подходящий.
- **Сбой питания во время копирования**: Что если питание пропадёт во время переноса данных? Роль должна использовать rsync с возможностью продолжения и проверять целостность после завершения.
- **Повторный запуск роли**: Что происходит при повторном выполнении на уже перенесённой системе? Роль должна обнаруживать что перенос уже выполнен и не выполнять destructive операции повторно.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST автоматически определять подключённый SSD диск среди блочных устройств
- **FR-002**: Система MUST создавать таблицу разделов GPT на SSD (для совместимости с современными системами)
- **FR-003**: Система MUST форматировать раздел SSD в файловую систему ext4 с опциями оптимизации для flash накопителей (noatime)
- **FR-004**: Система MUST копировать все файлы из корневой файловой системы microSD на SSD включая права доступа, владельцев, временные метки и символические ссылки
- **FR-005**: Система MUST определять UUID отформатированного SSD раздела для использования в конфигурационных файлах
- **FR-006**: Система MUST обновлять файл `/etc/fstab` на скопированной системе для монтирования корня по UUID SSD раздела
- **FR-007**: Система MUST сохранять монтирование `/boot` раздела на microSD карте в fstab
- **FR-008**: Система MUST обновлять файл `/boot/cmdline.txt` на microSD для указания параметра `root=UUID=` с UUID SSD раздела
- **FR-009**: Система MUST проверять свободное место на SSD перед началом копирования
- **FR-010**: Система MUST предоставлять опцию отката или восстановления при ошибке переноса
- **FR-011**: Ansible роль MUST быть идемпотентной - повторный запуск не должен выполнять destructive operations если перенос уже завершён
- **FR-012**: Ansible роль MUST поддерживать переменные для указания целевого SSD устройства (например, `ssd_device: /dev/sda`)
- **FR-013**: Система MUST обеспечивать подтверждение от пользователя перед форматированием если SSD содержит данные
- **FR-014**: Система MUST монтировать SSD во временную директорию для операций копирования и настройки
- **FR-015**: Система MUST использовать rsync с опциями сохранения атрибутов (-axHAWX) для точного копирования файловой системы

### Key Entities

- **Raspberry Pi Node**: Узел кластера на котором выполняется перенос системы, имеет характеристики (IP адрес, модель Pi, текущий boot носитель)
- **SSD Device**: Физическое устройство хранения подключённое через USB, имеет характеристики (имя устройства /dev/sdX, объём, текущее состояние)
- **Root Filesystem**: Корневая файловая система Linux содержащая ОС, приложения, данные, имеет атрибуты (точка монтирования /, файловая система ext4, текущее расположение)
- **Boot Partition**: Раздел загрузчика находящийся на microSD карте, содержит ядро и конфигурацию загрузки (cmdline.txt, config.txt)
- **UUID Partition**: Уникальный идентификатор раздела используемый для надёжного монтирования, генерируется при форматировании

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: После выполнения роли Raspberry Pi успешно загружается с SSD как корневой файловой системы за время не превышающее обычную загрузку (подтверждается проверкой `df -h` что `/` смонтирован с `/dev/sdX1`)
- **SC-002**: Время операций чтения/записи на файловой системе уменьшается минимум на 50% по сравнению с microSD (измеряется benchmark тестами до и после переноса)
- **SC-003**: Раздел `/boot` остаётся на microSD карте и не изменяется в процессе переноса (проверяется что `/boot` смонтирован с `/dev/mmcblk0p1`)
- **SC-004**: Перенос системы успешно выполняется на хосте leha (10.0.1.104) без ручного вмешательства после подтверждения начальных условий
- **SC-005**: Ansible роль является идемпотентной - повторный запуск на уже перенесённой системе не выполняет destructive operations (форматирование, копирование) и возвращает статус unchanged
- **SC-006**: Роль выполняется за менее чем 30 минут для типичной системы с использованием до 20GB на microSD (время copying phase зависит от скорости USB и объёма данных)
- **SC-007**: Все права доступа, владельцы файлов и символические ссылки сохраняются после переноса (проверяется сравнением атрибутов ключевых системных файлов и директорий)
- **SC-008**: При недостатке места на SSD роль прерывается с сообщением об ошибке до начала форматирования (100% предотвращается потеря данных из-за нехватки места)

## Scope & Boundaries *(mandatory)*

### In Scope

- Перенос корневой файловой системы Raspberry Pi OS с microSD на SSD
- Настройка автоматической загрузки с SSD (boot раздел остаётся на microSD)
- Создание переиспользуемой Ansible роли для автоматизации переноса
- Идемпотентность роли для безопасного повторного запуска
- Проверка целостности перенесённых данных
- Поддержка Raspberry Pi 4/5 с USB boot capability

### Out of Scope

- Перенос вручную без Ansible (документация может существовать отдельно)
- Поддержка других файловых систем кроме ext4
- RAID или LVM конфигурации на SSD
- Миграция данных с существующего SSD на новый SSD
- Автоматическое создание резервных копий microSD перед переносом
- Поддержка других дистрибутивов Linux кроме Raspberry Pi OS
- GUI или интерактивный мастер настройки (роль полностью автоматическая с prompt только для критических подтверждений)

## Assumptions & Dependencies *(mandatory)*

### Assumptions

- Raspberry Pi работает под управлением Raspberry Pi OS (Debian-based)
- microSD карта содержит работающую систему с boot и rootfs разделами
- SSD подключён через USB порт и определяется системой как блочное устройство (/dev/sdX)
- Raspberry Pi 4/5 с поддержкой USB boot (или уже настроен boot из USB)
- У пользователя есть sudo/root права на целевом хосте
- Ansible контроллер имеет SSH доступ к целевому хосту с правами escalation
- Объём SSD равен или превышает текущее использование на microSD

### Dependencies

- Наличие установленного Ansible на контроллере (версии 2.9 или выше)
- Целевой хост должен быть доступен по SSH
- На целевом хосте должны быть установлены утилиты: lsblk, fdisk, mkfs.ext4, rsync, blkid
- Наличие подключения к интернету для установки пакетов если требуется (apt)
- Наличие inventory файла с определением хоста leha (10.0.1.104) и необходимых групп переменных
