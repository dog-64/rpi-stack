# Feature Specification: Установка K3s на Raspberry Pi стеке

**Feature Branch**: `002-k3s-install`
**Created**: 2026-02-15
**Status**: Draft
**Input**: User description: "Установка k3s на хостах Raspberry Pi стека. Сначала вручную на leha (главный хост кластера), потом на sema. Затем создание Ansible роли."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Ручная установка K3s server на leha (Priority: P1)

Администратор вручную устанавливает K3s в режиме server (control plane) на хосте leha (Raspberry Pi 5, 8GB RAM, IP 10.0.1.104). Это главный узел кластера, на котором будет работать API-сервер Kubernetes. Ручная установка позволяет понять процесс перед автоматизацией и проверить совместимость с конкретным оборудованием.

**Why this priority**: Без работающего control plane невозможны все последующие шаги. Ручная установка даёт понимание процесса и уверенность в работоспособности на конкретном железе.

**Independent Test**: Можно полностью проверить одной командой: `kubectl get nodes` с хоста leha должен показать один узел в статусе Ready.

**Критерии успеха перед началом**:
- Хост leha доступен по SSH
- Достаточно свободного места на диске (минимум 2 GB)
- Архитектура arm64 подтверждена (`uname -m` = aarch64)

**Acceptance Scenarios**:

1. **Given** чистый хост leha с Ubuntu/Raspberry Pi OS, **When** администратор выполняет установку K3s server, **Then** процесс K3s запущен и API-сервер доступен на порту 6443
2. **Given** K3s server установлен на leha, **When** администратор выполняет `kubectl get nodes`, **Then** узел leha отображается со статусом Ready
3. **Given** K3s server работает, **When** администратор выполняет `kubectl get pods -A`, **Then** все системные поды (coredns, local-path-provisioner, metrics-server) в статусе Running
4. **Given** K3s server работает, **When** администратор перезагружает хост, **Then** K3s автоматически запускается после загрузки

---

### User Story 2 - Подключение sema как worker-узла (Priority: P2)

Администратор вручную устанавливает K3s в режиме agent (worker) на хосте sema (Raspberry Pi 5, 8GB RAM, IP 10.0.1.33) и присоединяет его к кластеру, управляемому leha. После этого кластер состоит из двух узлов и способен запускать рабочие нагрузки.

**Why this priority**: Кластер из одного узла не обеспечивает отказоустойчивости и распределения нагрузки. Добавление второго узла делает кластер функциональным.

**Independent Test**: `kubectl get nodes` с leha показывает два узла (leha и sema) в статусе Ready.

**Критерии успеха перед началом**:
- leha уже работает как K3s server (User Story 1 выполнена)
- Хост sema доступен по SSH
- Сетевая связность между leha и sema подтверждена (ping, порт 6443)
- Токен кластера получен с leha

**Acceptance Scenarios**:

1. **Given** K3s server работает на leha, **When** администратор устанавливает K3s agent на sema с указанием URL и токена сервера, **Then** sema появляется в списке узлов кластера
2. **Given** sema подключена к кластеру, **When** администратор выполняет `kubectl get nodes` на leha, **Then** оба узла (leha, sema) отображаются со статусом Ready
3. **Given** кластер из двух узлов работает, **When** администратор создаёт тестовый pod, **Then** pod запускается на одном из узлов и переходит в статус Running
4. **Given** кластер работает, **When** администратор перезагружает sema, **Then** после загрузки sema автоматически подключается к кластеру

---

### User Story 3 - Создание Ansible роли для автоматизации установки K3s (Priority: P3)

После успешной ручной установки администратор создаёт Ansible роли (k3s_server и k3s_agent) для автоматической установки K3s на любом хосте стека. Роли должны быть идемпотентными и повторно запускаемыми. В дальнейшем можно будет добавить в кластер хосты motya и osya.

**Why this priority**: Автоматизация нужна для воспроизводимости, масштабирования на оставшиеся хосты и восстановления после сбоев. Но сначала нужно проверить процесс вручную.

**Independent Test**: Запуск `ansible-playbook` на чистых хостах приводит к работающему кластеру K3s. Повторный запуск не вносит изменений (идемпотентность).

**Критерии успеха перед началом**:
- Ручная установка на leha и sema прошла успешно (US1, US2)
- Процесс установки задокументирован со всеми командами и параметрами
- Известны все необходимые предусловия (пакеты, настройки ядра, сети)

**Acceptance Scenarios**:

1. **Given** Ansible роли созданы, **When** администратор запускает playbook на чистых хостах, **Then** K3s кластер разворачивается полностью автоматически
2. **Given** K3s кластер уже развёрнут, **When** администратор повторно запускает playbook, **Then** Ansible не вносит изменений (идемпотентность, 0 changed)
3. **Given** роли готовы, **When** администратор добавляет новый хост (motya или osya) в inventory как worker, **Then** новый узел присоединяется к кластеру после запуска playbook

---

### User Story 4 - Валидация и диагностика кластера (Priority: P4)


Администратор имеет набор проверок для быстрой диагностики состояния кластера: здоровье узлов, системных подов, сетевой связности между узлами, и температуры/нагрузки Raspberry Pi.

**Why this priority**: Без инструментов диагностики невозможно быстро выявлять и устранять проблемы. Важно, но зависит от работающего кластера.

**Independent Test**: Скрипт верификации запускается и выдаёт отчёт о состоянии кластера с чётким статусом PASS/FAIL.

**Acceptance Scenarios**:

1. **Given** кластер K3s работает, **When** администратор запускает скрипт проверки, **Then** выводится отчёт с состоянием каждого узла, подов, и сетевой связности
2. **Given** один из узлов выключен, **When** администратор запускает проверку, **Then** отчёт чётко указывает какой узел недоступен и какие сервисы затронуты

---

### User Story 5 - Ansible роль удаления K3s (Priority: P5)

Администратор может полностью удалить K3s с любого хоста кластера через Ansible роль. Это необходимо для чистой переустановки, отката при проблемах или вывода узла из кластера.

**Why this priority**: Делается в последнюю очередь. Важно для безопасности эксплуатации, но не блокирует основную функциональность.

**Independent Test**: После запуска роли uninstall на хосте, K3s полностью удалён, все процессы остановлены, конфигурация очищена.

**Acceptance Scenarios**:

1. **Given** K3s agent работает на хосте, **When** администратор запускает роль uninstall, **Then** K3s полностью удалён (процессы, бинарники, конфигурация, данные)
2. **Given** K3s server работает на хосте, **When** администратор запускает роль uninstall, **Then** K3s server полностью удалён, включая SQLite базу и токены
3. **Given** K3s удалён с хоста, **When** администратор повторно запускает роль установки, **Then** K3s устанавливается чисто без конфликтов с предыдущей установкой

---

### Edge Cases

- Что происходит, если leha недоступен (сбой control plane)? Agent-узлы продолжают работать с уже запущенными подами, но новые поды не могут быть запланированы
- Что происходит при потере сетевого соединения между leha и sema? Agent помечается как NotReady, поды перепланируются после таймаута (5 минут по умолчанию)
- Что если на хосте уже установлена старая версия K3s? Скрипт установки обновляет K3s до целевой версии
- Что если SD-карта/SSD заполнена? K3s не сможет запуститься, нужно минимум 2 GB свободного места
- Что если температура процессора Pi превышает 80C? CPU throttling замедлит работу K3s, нужен мониторинг температуры
- Что происходит при отключении питания во время установки? Повторный запуск скрипта установки должен корректно завершить процесс (идемпотентность)
- Что если на рабочей станции уже есть ~/.kube/config с другими кластерами? Категорически запрещено перезаписывать — используется отдельный файл kubeconfig или безопасное добавление контекста

## Clarifications

### Session 2026-02-15

- Q: Какой уровень безопасности кластера? → A: Базовый — API-сервер слушает только локальную сеть, kubeconfig с ограниченными правами, firewall между узлами
- Q: Откуда управлять кластером через kubectl? → A: Только с Mac (рабочей станции). КРИТИЧНО: существующий kubeconfig (~/.kube/config) НЕ ДОЛЖЕН быть перезаписан — использовать отдельный файл или безопасное слияние контекстов
- Q: Конфликт swap: setup-cluster.yml ставит swappiness=10, K3s требует полного отключения? → A: Обновить setup-cluster.yml — swappiness=0, swapoff -a, убрать из fstab
- Q: Нужна ли процедура удаления/отката K3s? → A: Да, создать Ansible роль uninstall (k3s-uninstall.sh + очистка). Делать в последнюю очередь (низший приоритет)
- Q: Фиксировать ли версию K3s? → A: Да, фиксировать в group_vars (например k3s_version: v1.31.4+k3s1), обновлять вручную

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Кластер ДОЛЖЕН поддерживать архитектуру arm64 (aarch64) на Raspberry Pi 4 и Pi 5. Версия K3s ДОЛЖНА быть зафиксирована в переменной Ansible (group_vars) и устанавливаться явно через INSTALL_K3S_VERSION
- **FR-002**: K3s server ДОЛЖЕН быть установлен на хосте leha (10.0.1.104) как единственный control plane узел
- **FR-003**: K3s agent ДОЛЖЕН быть установлен на хосте sema (10.0.1.33) как первый worker-узел
- **FR-004**: Кластер ДОЛЖЕН использовать встроенный Flannel CNI (vxlan backend) как рекомендовано для Raspberry Pi
- **FR-005**: K3s ДОЛЖЕН запускаться автоматически после перезагрузки хостов (systemd service)
- **FR-006**: Установка ДОЛЖНА отключать неиспользуемые компоненты: Traefik и ServiceLB (для экономии ресурсов Pi)
- **FR-007**: На хостах ДОЛЖНА быть полностью отключена подкачка (swap): swapoff -a, удаление записей из fstab, vm.swappiness=0. Существующий playbook setup-cluster.yml ДОЛЖЕН быть обновлён (текущее значение swappiness=10 конфликтует с требованием K3s)
- **FR-008**: На хостах ДОЛЖНЫ быть загружены необходимые модули ядра: overlay, br_netfilter
- **FR-009**: Sysctl параметры ДОЛЖНЫ быть настроены: net.bridge.bridge-nf-call-iptables=1, net.ipv4.ip_forward=1
- **FR-010**: kubeconfig ДОЛЖЕН быть настроен на рабочей станции администратора (Mac) для удалённого управления кластером через kubectl
- **FR-010a**: Существующий kubeconfig (~/.kube/config) на рабочей станции ЗАПРЕЩЕНО перезаписывать. Новый контекст кластера ДОЛЖЕН быть добавлен как отдельный файл (например, ~/.kube/rpi-k3s.yaml) или безопасно слит через KUBECONFIG env или `kubectl config merge`
- **FR-010b**: Ansible роль ДОЛЖНА предусматривать безопасное копирование kubeconfig с leha на рабочую станцию с заменой server-адреса (127.0.0.1 → 10.0.1.104) без затрагивания существующих конфигов
- **FR-011**: Ansible роль k3s_server ДОЛЖНА устанавливать и настраивать K3s в режиме server
- **FR-012**: Ansible роль k3s_agent ДОЛЖНА устанавливать K3s в режиме agent и подключать к серверу
- **FR-013**: Ansible роли ДОЛЖНЫ быть идемпотентными — повторный запуск не вносит изменений
- **FR-014**: Ansible роль prerequisites ДОЛЖНА готовить хосты (пакеты, модули ядра, sysctl, swap)
- **FR-015**: Процесс установки ДОЛЖЕН быть задокументирован с командами для каждого шага
- **FR-016**: API-сервер K3s ДОЛЖЕН быть привязан к локальной сети (--bind-address=10.0.1.104, --advertise-address=10.0.1.104), а не слушать на 0.0.0.0
- **FR-017**: kubeconfig ДОЛЖЕН иметь ограниченные права доступа (chmod 600) и быть доступен только пользователю-администратору
- **FR-018**: Межузловой трафик кластера ДОЛЖЕН быть ограничен необходимыми портами: 6443 (API), 8472 (Flannel VXLAN), 10250 (kubelet)

### Key Entities

- **K3s Server (Control Plane)**: Главный узел кластера, запускает API-сервер, scheduler, controller-manager. Хост: leha. Хранит состояние кластера в SQLite (встроенное хранилище K3s)
- **K3s Agent (Worker)**: Рабочий узел, получает задания от server, запускает пользовательские контейнеры. Хост: sema (первый), потенциально motya и osya
- **Cluster Token**: Секретный токен для аутентификации agent-узлов при присоединении к кластеру. Генерируется при установке server, хранится в /var/lib/rancher/k3s/server/node-token
- **kubeconfig**: Файл конфигурации для доступа к API кластера. Хранится в /etc/rancher/k3s/k3s.yaml на server-узле

## Assumptions

- Все хосты работают на Raspberry Pi OS (Debian-based) с ядром 64-bit (arm64)
- Хосты leha и sema уже настроены базовым playbook setup-cluster.yml (пакеты, hostname, /etc/hosts, timezone)
- Сетевая связность между хостами обеспечена через локальную сеть (10.0.1.0/24)
- SSH доступ от рабочей станции администратора ко всем хостам настроен
- Используется Flannel CNI (встроенный в K3s) — Cilium слишком ресурсоёмкий для Raspberry Pi
- Версия K3s фиксируется в переменной Ansible (group_vars) и обновляется вручную для контролируемых апгрейдов
- HA (высокая доступность) control plane не требуется на данном этапе — один server-узел достаточен для обучения и разработки
- Хосты motya и osya будут добавлены в кластер как worker-узлы позднее, используя созданные Ansible роли

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Команда `kubectl get nodes` на leha возвращает все узлы кластера со статусом Ready в течение 5 минут после установки
- **SC-002**: Тестовый pod (nginx) запускается и становится доступен в кластере в течение 2 минут после создания
- **SC-003**: После перезагрузки любого узла, он автоматически возвращается в кластер в статусе Ready в течение 3 минут
- **SC-004**: Ansible playbook на чистых хостах завершает полную установку кластера за один запуск без ошибок
- **SC-005**: Повторный запуск Ansible playbook на уже настроенных хостах показывает 0 changed tasks (идемпотентность)
- **SC-006**: Системные поды K3s (coredns, local-path-provisioner, metrics-server) работают стабильно без перезапусков в течение 1 часа
- **SC-007**: Потребление RAM на каждом узле кластером K3s не превышает 1 GB в idle-состоянии (без пользовательских нагрузок)
